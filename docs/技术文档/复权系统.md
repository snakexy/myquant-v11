## 2. 核心模块详解

### 2.1 AdjustmentFactorService（复权因子服务）

**文件**: `backend/src/myquant/core/market/services/adjustment_factor_service.py`

#### 职责
- 管理复权因子表的计算和缓存
- 支持两种复权方式的因子计算
- 提供快速查表应用功能

#### 关键方法

| 方法 | 说明 | 性能 |
|------|------|------|
| `get_factor_table(symbol, adjust_type)` | 获取复权因子表（带两级缓存） | 首次 200-500ms，后续 <5ms |
| `apply_factors(df, factor_table)` | 应用复权因子到K线数据 | O(n) 查表，<5ms/1000条 |
| `calculate_front_factors(xdxr_data)` | 计算前复权累积因子 | 约 100-300ms |
| `calculate_front_ratio_factors(xdxr_data)` | 计算等比复权独立因子 | 约 50-100ms |

#### 缓存层级

```python
# L1 内存缓存
self._factor_cache = TTLCache(maxsize=500, ttl=7 * 24 * 3600)

# L2 文件缓存（Binary Pickle，比 JSON 快 5-10 倍）
data/xdxr_cache/{symbol}/factors_{adjust_type}.pkl
```

#### 使用示例

```python
from myquant.core.market.services.adjustment_factor_service import get_adjustment_factor_service

# 获取服务实例
service = get_adjustment_factor_service()

# 获取因子表（自动使用缓存）
factor_table = service.get_factor_table('600519.SH', 'front')

# 应用到K线数据
df_adjusted = service.apply_factors(df_raw, factor_table)
```

### 2.2 XDXR 数据缓存（高性能读取）

**文件**: `backend/src/myquant/core/market/services/seamless_service.py`

#### 优化点

| 优化项 | 实现方式 | 性能提升 |
|--------|---------|---------|
| 二进制格式 | Pickle 替代 JSON | 5-10x |
| 目录创建优化 | 类级别标记，只检查一次 | 减少 IO |
| 文件检查优化 | 合并 exists + stat 调用 | 减少 IO |
| 格式自动回退 | .pkl 不存在则尝试 .json | 兼容旧数据 |

#### 存储结构

```
data/xdxr_cache/
├── 000001_SZ/
│   ├── 000001_SZ.pkl          # 二进制格式（优先）
│   ├── 000001_SZ.json         # JSON格式（兼容）
│   ├── factors_front.pkl      # 前复权因子表
│   └── factors_front_ratio.pkl # 等比复权因子表
```

#### 文件读取性能对比

| 格式 | 文件大小 | 读取时间 | 适用场景 |
|------|---------|---------|---------|
| JSON | 100KB | 10-20ms | 调试、兼容 |
| Pickle | 65KB | 1-3ms | 生产环境（推荐） |

### 2.3 混合模式复权

**核心逻辑**: `seamless_service.py:_apply_adjustment()`

#### 策略

```python
if period == '1d':
    # 日线：使用传统前复权（累积因子）
    actual_adjust_type = 'front'
else:
    # 分钟线：使用等比复权（独立因子）
    actual_adjust_type = 'front_ratio'
```

#### 两种复权方式对比

| 复权方式 | 计算方式 | 适用周期 | 优点 | 缺点 |
|---------|---------|---------|------|------|
| 前复权(front) | 累积因子：从最新往回乘 | 日线 | 最新价不变，直观 | 需要完整历史 |
| 等比复权(front_ratio) | 独立因子：1/(1+送股率+配股率) | 分钟线 | 不依赖历史完整 | 最新价会变动 |

#### 为什么分钟线不能用前复权？

```
前复权计算：需要找到最新的除权日，往回累积计算

日线数据：通常有几年历史，能覆盖所有除权日
分钟线数据：通常只有最近 1-3 个月，可能不包含除权日

结果：使用累积因子时，找不到最近的除权日，导致复权错误

等比复权：每个除权日独立计算，不依赖数据范围，适合分钟线
```

### 2.4 自选股预热机制

**文件**: `backend/src/myquant/core/market/services/watchlist_prewarmer.py`

#### 职责
- 管理用户自选股列表
- 在后台预加载自选股的复权因子
- 自选股切换时保证 <5ms 响应

#### 使用方式

```python
from myquant.core.market.services.watchlist_prewarmer import (
    set_watchlist,
    warmup_watchlist,
    get_watchlist_prewarmer
)

# 设置自选股列表（用户修改自选股后调用）
set_watchlist(['000001.SZ', '600519.SH', '000858.SZ'])

# 应用启动时预热
warmup_watchlist()

# 或在后台线程异步预热
import threading
threading.Thread(target=warmup_watchlist, daemon=True).start()
```

#### 预热效果

| 场景 | 首次加载时间 | 说明 |
|------|-------------|------|
| 未预热 | 200-500ms | 需要计算因子表 |
| 已预热 | <5ms | 内存缓存命中 |

## 3. 回测场景最佳实践

### 3.1 回测复权流程

```python
# 1. 定义回测股票池
stock_pool = ['000001.SZ', '600519.SH', ...]  # 几十~几百只

# 2. 批量预热（回测开始前一次性执行）
from myquant.core.market.services.watchlist_prewarmer import WatchlistPrewarmer

prewarmer = WatchlistPrewarmer()
prewarmer.set_watchlist(stock_pool)
prewarmer.warmup()  # 异步预热所有股票的两种复权因子

# 3. 加载K线并复权（每只 <5ms）
service = get_adjustment_factor_service()

for symbol in stock_pool:
    # 加载原始K线
    df_raw = load_kline(symbol, period='1d', ...)

    # 获取因子表（内存命中，<1ms）
    factor_table = service.get_factor_table(symbol, 'front')

    # 应用复权（<5ms）
    df_adjusted = service.apply_factors(df_raw, factor_table)

    # 开始回测计算...
```

### 3.2 性能指标

| 操作 | 时间 | 说明 |
|------|------|------|
| 预热 100 只股票 | 20-50 秒 | 一次性，异步执行 |
| 单只股票复权 | <5ms | 内存缓存命中 |
| 1000只股票 × 5年日线 | 5-10 秒 | 批量处理 |

### 3.3 注意事项

1. **只使用前复权**：回测通常只关心相对价格变化，前复权足够
2. **不需要后复权**：除非要精确匹配历史交易金额（极少需要）
3. **不需要持久化复权后数据**：实时应用足够快，避免存储冗余

## 5. 故障排查

### 5.1 常见问题

#### Q1: 首次加载还是很慢（>500ms）
- **检查**: 是否从网络获取 XDXR 数据（而非文件缓存）
- **解决**: 确保 `data/xdxr_cache/` 目录有写入权限，且已生成 .pkl 文件

#### Q2: 复权结果不对（价格异常）
- **检查**: XDXR 数据格式是否匹配（不同数据源字段不同）
- **解决**: 查看日志 `[FactorCalc]` 是否有警告，确认 category 字段存在

#### Q3: 内存占用过高
- **检查**: `TTLCache` 的 `maxsize` 设置
- **解决**: 调整 `AdjustmentFactorService._factor_cache` 的 maxsize（默认 500）

### 5.2 日志关键词

```bash
# 查看复权相关日志
tail -f data/logs/app.log | grep -E "(Factor|XDXR|复权|预热)"

# 关键日志标记
[FactorCache]     # 因子缓存命中/未命中
[FactorCalc]      # 因子计算过程

[预热]            # 自选股预热进度
```

## 6. 附录：复权算法详解

### 6.1 前复权计算（日线用）

```python
def calculate_front_factors(xdxr_data):
    """
    算法步骤：
    1. 筛选除权除息记录 (category=1)
    2. 按日期排序（从新到旧）
    3. 计算每个除权日的单日复权因子：
       factor = (前收 + 配股×配股价 - 分红) / (前收 × (1 + 送股 + 配股))
    4. 累积因子：从最新往旧累积乘
    5. 展开为每日因子表
    """
```

### 6.2 等比复权计算（分钟线用）

```python
def calculate_front_ratio_factors(xdxr_data):
    """
    算法步骤：
    1. 筛选除权除息记录
    2. 按日期排序（从新到旧）
    3. 计算每个除权日的独立因子：
       factor = 1 / (1 + 送股率 + 配股率)
    4. 同样累积（但因子计算方式不同，不依赖除权前价格）
    5. 展开为每日因子表
    """
```

### 6.3 因子应用

```python
def apply_factors(df, factor_table):
    """
    O(n) 查表操作：
    1. 提取日期字符串: df['date'] = df['datetime'].dt.strftime('%Y-%m-%d')
    2. 查表获取因子: df['factor'] = df['date'].map(factor_table).fillna(1.0)
    3. 应用到价格: df['open'] *= df['factor'], ...
    """
```
