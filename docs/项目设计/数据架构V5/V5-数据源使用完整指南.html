L0-L5 数据分层架构

🏗️ 分层设计原则

MyQuant采用L0-L5分层数据架构，根据数据类型自动选择最优数据源和获取方式，实现性能最大化。

每一层都有明确的数据内容、获取方式、数据源优先级和适用场景。

L0

订阅缓存（实时推送）

🚀 <1ms
项目 内容
数据内容 已订阅股票的实时行情缓存
获取方式 实时: 订阅推送(<1ms) | 历史: 直接查询API(无需订阅)
数据源 XtQuant (订阅上限300股/实例 × 2实例 = 600只)
注意：订阅用于实时推送，K线数据可直接查询获取
适用场景 高频监控（每秒更新）、实时价格提醒、实时K线图
L1

实时快照

⚡ <1ms / ~10ms
项目 内容
数据内容 股票当前价格、成交量等实时信息（5档行情）
获取方式 优先: 订阅推送(<1ms) | 备选: 批量API(~10ms)
数据源 TdxQuant (0.60ms, 最快) → XtQuant (0.90ms) → PyTdx2 (10-19ms) → LocalDB
适用场景 获取实时价格、实时行情监控、盘口观察
L2

历史摘要

📊 7-50ms

获取方式 优先: 本地数据库 | 备选: API批量获取
数据源 QLib LocalDB (7-10ms) → TdxQuant (盘后最快, 注意:盘中不支持分钟K线) → XtQuant (盘中分钟K线首选) → PyTdx2
适用场景 快速查看股票概况、技术指标分析、短期趋势判断

完整数据

数据源 LocalDB (~5ms) → TdxQuant (0.60ms/单只, 最快) → XtQuant (0.90ms/单只) → PyTdx2 (14ms/单只)
适用场景 K线图表显示、技术分析回测、策略开发、日内交易

XtQuant - 当天分钟线专家

核心优势

  • 当天分钟线: 0.90ms缓存 - 交易时间最优
  • 订阅功能: 300股上限, <1ms推送（注意：订阅用于实时推送，不订阅也可以直接获取K线数据）
  • L1快照: 720ms下载 + 6ms缓存

⚠️ 关键限制

❌ 不支持板块数据 - 板块业务必须用 TdxQuant/PyTdx2

⚠️ 读取本地缓存 - get_market_data_ex 读取 QMT 本地数据，非真正在线

⚠️ 仅交易+盘后可用 - 周末/节假日无法使用

⚠️ 重要说明: get_market_data_ex 的真实行为：

  • 读取QMT本地缓存，并非真正的"在线获取"
  • 数据范围取决于QMT本地数据库的存储情况
  • 获取当天数据：需在交易时间让QMT自动更新，或先调用 download_history_data()

本地缓存读取能力：

  • 日K: 750条 (3年) - 9ms
  • 5分钟K: 5532条 (约5个月) - 15ms - 本地缓存上限
  • 1分钟K: 2500条 (约2周) - 12ms - 本地缓存上限

订阅机制详解

特性 XtQuant TdxQuant
订阅上限 300股 100股
推送延迟 <1ms 事件推送 5-6ms 轮询
推送方式 有变化才推送 定期轮询

💡 重要说明：订阅 vs 直接查询

订阅的作用：实时推送行情数据（<1ms延迟），适用于高频监控场景

直接查询K线：XtQuant 不需要订阅也可以直接获取历史K线数据，包括当日分钟线

使用建议：

  • 需要实时推送（<1秒更新）→ 使用订阅
  • 只需获取K线数据（任何周期）→ 直接查询，无需订阅
  • 订阅上限300股/实例，超出限制请使用TdxQuant或PyTdx2

数据刷新机制 🔄

刷新机制核心原则

V5 场景化服务采用多维度刷新策略，确保数据新鲜度与性能的平衡：

  • 增量更新: 只获取缺失数据，避免重复请求
  • 智能缺口检测: 自动识别数据缺口
  • 时间自适应: 根据交易时间选择最优数据源
  • 缓存一致性: 刷新后自动更新缓存

按数据类型的刷新策略

L0/L1 实时数据

刷新方式: 订阅 + 轮询

  • L0: 订阅推送，有变化立即更新
  • L1: 5秒轮询刷新 Redis
  • 缓存: TTL 60秒自动过期

数据源: TdxQuant(0.60ms) → XtQuant → PyTdx2

L2/L3 历史数据

刷新方式: 增量更新

  • 日K线: 盘后增量更新当日数据
  • 周/月K: 自动聚合计算
  • 检测: 每日自动检测缺口

数据源: QLib本地 → PyTdx2在线

L5 板块数据

刷新方式: 定时更新

  • 交易时间: TdxQuant 30分钟更新
  • 非交易时间: PyTdx2 按需获取
  • 板块分类: 每日盘后全量更新

数据源: TdxQuant(586板块) → PyTdx2

L4 财务数据

刷新方式: 季度更新

  • 财报季: 自动触发更新
  • 检测: 对比发布日期
  • 补充: 历史数据回填

数据源: TdxQuant(46字段) → XtQuant

常见问题FAQ

Q1: 为什么不能用TdxQuant获取当天分钟线？

A: TdxQuant API的限制。`get_kline()`只支持历史K线，不支持当天分钟级别的实时数据。

正确做法:

  • 交易时间: XtQuant.get_market_kline() (0.90ms)
  • 非交易时间: PyTdx2.get_kline() (10-19ms)

Q2: 为什么不能用XtQuant获取板块数据？

A: XtQuant完全不支持板块数据，这是功能限制，不是配置问题。

正确做法:

  • 交易时间: TdxQuant.get_sectors() (6.99ms, 586板块)
  • 非交易时间: PyTdx2.get_sectors() (~15ms)

Q3: 非交易时间应该用什么数据源？

A: 只能用PyTdx2（24/7在线数据源）或QLib（本地数据库）

数据源 状态
TdxQuant ❌ 需要通达信终端运行
XtQuant ❌ 需要QMT运行
PyTdx2 ✅ 24/7在线
QLib ✅ 24/7本地

Q4: 板块数据为什么特别重要？

板块数据在风险预测中的关键作用 ⭐

  • 板块轮动分析 - 识别热点板块，预测板块切换时机
  • 系统性风险监控 - 板块整体涨跌，市场情绪指标
  • 投资组合优化 - 板块配置权重，风险敞口控制

没有板块数据的风险预测是不完整的！

股票数量 数据源 原因
<100 TdxQuant 🔥 0.60ms最快
100-300 XtQuant 6ms缓存
>300 PyTdx2 分批获取
