# MyQuant V5 架构规范

## 核心设计理念：场景化服务

V5 不是通用分层架构，而是**按业务场景专精**。每个服务只做一件事，内部自己决定用哪个数据源。

### 数据流（必须遵守）
```
前端 → API路由层(api/*) → Service层(core/*/services/) → Adapter层(core/*/adapters/) → 数据源
```
- 前端只能调用 `/api/*` 路由，不能跳层
- API 层只做参数校验和路由，**不包含业务逻辑**
- Service 层通过 `get_adapter()` 获取 Adapter，**不能直接实例化**

---

## V5 四大场景服务

| 服务 | 主要场景 | 核心数据源 |
|------|----------|------------|
| **MonitoringService** | 热点发现、异常检测、实时行情监控 | TdxQuant(交易时间) → PyTdx2 |
| **IncrementalService** | 填补缺失数据、数据完整性维护 | XtQuant(交易时间) → PyTdx2 |
| **KlineService** | 实时K线推送、分钟聚合、K线图表 | XtQuant + PyTdx2 + QLib |
| **ConversionService** | 批量数据转换（Tdx→Qlib）、历史迁移 | 直接读本地文件，不用在线 |

---

## L0-L5：数据类型分类（不是服务层级！）

| 层级 | 数据类型 | 典型用途 |
|------|----------|----------|
| L0 | 订阅推送（实时 tick） | XtQuant 300股 / TdxQuant 100股 |
| L1 | 快照（当前价格） | TdxQuant 0.60ms（交易时间最快） |
| L2 | 当天分钟线 | XtQuant 0.90ms（TdxQuant不支持！） |
| L3 | 历史K线 | QLib 7-10ms / PyTdx2 在线 |
| L4 | 财务/基本面数据 | PyTdx2 / TdxQuant |
| L5 | 板块数据 | TdxQuant 6.99ms 586板块（XtQuant不支持！） |

---

## 双层路由设计

每个场景服务内部实现双层路由：

**第一层：数据类型区分**
- 板块数据（L5）→ 板块专用路由（XtQuant完全不支持）
- 非板块数据 → 通用路由

**第二层：时间段 + 数量**
- 交易时间：TdxQuant(快照<100股) / XtQuant(分钟线/100-300股) / PyTdx2(>300股/兜底)
- 非交易时间：PyTdx2（唯一可用）

---

## 四大数据源能力速查

| 数据源 | 最擅长 | 关键限制 | 可用时间 |
|--------|--------|----------|----------|
| **TdxQuant** | L1快照(0.60ms) + L5板块(6.99ms, 586板块) | **不支持当天分钟线(L2)** | 仅交易时间 |
| **XtQuant** | L2当天分钟线(0.90ms) + L0订阅(300股) | **不支持板块(L5)** | 交易+盘后 |
| **PyTdx2** | 24/7全品类兜底 | 响应稍慢(10-19ms) | **24/7** |
| **QLib** | L3历史K线(7-10ms) 本地最快 | 不支持快照/板块/订阅 | **24/7** |

---

## 关键禁忌

- ❌ 用 TdxQuant 获取当天分钟线（不支持，直接报错）
- ❌ 用 XtQuant 获取板块数据（不支持，代码映射不兼容）
- ❌ 非交易时间用 TdxQuant（服务不可用）
- ❌ 盘后用 TdxQuant 获取板块（只有昨天历史，应用 PyTdx2 真在线）
