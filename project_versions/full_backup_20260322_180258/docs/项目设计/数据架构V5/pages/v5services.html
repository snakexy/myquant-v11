V5 场景化服务架构

🔑 关键架构变革 - V5 替代 V4 统一数据管理服务

V4 架构问题: 单一 UnifiedDataManager (3777行) 管理所有数据获取逻辑

  • ❌ 代码臃肿，维护困难
  • ❌ 场景混杂，互相影响
  • ❌ 扩展性差，新增功能需修改核心

V5 解决方案: 场景化服务 - 每个服务专注一个业务场景

  • ✅ 代码模块化 (~500行/服务)
  • ✅ 独立测试，互不影响
  • ✅ 扩展性强，添加新服务不影响现有

V5 vs V4 架构对比

对比项 V4 统一管理器 V5 场景化服务
设计理念 一个服务管所有 每个服务专精一个场景
代码组织 3777行单文件 ~500行/服务模块
维护难度 困难（互相影响） 独立测试
扩展性 修改核心文件 添加新服务模块

四大场景服务定位

服务 主要场景 数据层级 优化策略
MonitoringService 热点发现、异常检测 L1 高速优先
IncrementalService 填补缺失数据 L3 智能检测
KlineService 实时K线推送 L0+L3 订阅+缓存
ConversionService 批量数据转换 L2-L3 直接读文件

V5 场景化服务层 路由层 适配器层

MonitoringService - 监控服务

定位: 热点发现、异常检测、实时行情监控

数据源路由:

  • 板块数据: TdxQuant(交易) → PyTdx2
  • 快照数据: TdxQuant → XtQuant → PyTdx2

使用场景: 热点板块发现、异常股票检测、实时行情监控

IncrementalService - 增量更新服务

定位: 填补缺失的交易日数据、在线获取最新数据

数据源路由:

  • 当天分钟线: XtQuant(交易) → PyTdx2
  • 历史K线: QLib → PyTdx2

使用场景: 数据完整性维护、缺失数据填补

KlineService - 实时K线服务

定位: 实时K线订阅推送、分钟级K线聚合

数据源路由:

  • 当天分钟线: XtQuant(交易) → PyTdx2
  • L0订阅: XtQuant(300股) → TdxQuant(100股)

使用场景: K线图表、实时K线推送

ConversionService - 数据转换服务

定位: 批量数据转换、历史数据迁移

数据源: 直接读本地 .day 文件，不使用在线API

使用场景: Tdx → Qlib 格式转换、批量数据迁移
